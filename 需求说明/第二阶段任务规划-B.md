# 第二阶段任务规划-B（基于当前实现基线）

## 1. 文档说明
- 对比来源：`第二阶段PLAN.md` 与 `第二阶段已完成任务-A.md`。
- 统计口径：以当前代码实际落地结果为基线，不再以原始 Phase 2 设想为唯一标准。
- 规划目标：在不进入第三阶段和研报线的前提下，补齐第二阶段剩余闭环，形成可稳定交付的“资讯链路生产化版本”。
- 结论摘要：第二阶段任务 B 定义为“资讯链路生产化收口 + 国内新闻源稳定化 + 简报与详情体验补齐”。

## 2. 当前基线与差异结论
### 2.1 已完成基线
- 资讯增强主链路已具备：标签落库、热榜、按 `sector/topic` 查询、`Ask AI`、Dashboard 资讯多视图。
- 日报能力已收敛到 `insights` 主链路，不再独立建设 `briefs` 模块。
- 第一阶段事件链路保持兼容，Dashboard 与 MarketView 的事件跳转闭环可用。

### 2.2 相对原始第二阶段计划的既成差异
- 原 `briefs` 独立模块路线已取消，改为 Insight 基座上的早报/晚报能力。
- Dashboard 资讯入口已不再适合机械沿用原始 `Events / Raw News / 7x24` 设想，信息架构需要按当前页面状态重新收口。
- 新闻源已从海外接口扩展到本地 `pobo-proxy` 主源，因此第二阶段后续任务必须纳入国内新闻源稳定性与时间语义校准。

### 2.3 规划原则
- B 阶段不重复 A 阶段已完成项。
- B 阶段优先补齐运行态问题、体验断点与验收缺口。
- B 阶段所有任务均以 Docker 运行态可验收为最终准入标准。

## 3. 任务分组
### 3.1 B1 资讯详情完整性
目标结果：
- 新闻详情打开后，摘要、标题、来源、时间、标签、相关标的信息完整可读。
- 详情页、弹窗、列表卡片之间字段口径一致，不再出现空摘要、无效摘要、时间错位、来源缺失等问题。

主要改动方向：
- 补齐新闻详情摘要兜底策略，保证在无正文摘要时可回退到标题或可用快照字段。
- 统一新闻详情字段来源，明确 `title / description / article_url / source / published_at / tickers / tags` 的优先级。
- 处理详情页与弹窗的空态、异常态和快照回退展示。

影响接口或页面：
- `GET /api/v1/news/{article_id}`
- Dashboard 新闻详情弹窗
- 资讯详情页

验收方式：
- 点击任意新闻卡片后，详情中的摘要区域不得为空白或无意义数值。
- 当原始新闻源字段缺失时，页面仍能显示可理解的最小信息。

### 3.2 B2 国内新闻源生产化
目标结果：
- `pobo-proxy` 作为主新闻源在 Docker 运行态下稳定可用。
- 新闻时间从源端、数据库到前端展示全链路语义一致。
- 主源故障时系统可控降级，不出现大量无效新闻、404 或时间错乱。

主要改动方向：
- 统一 `pobo-proxy` 时间语义，明确源时间按上海时区解释，再转换为系统内部 UTC。
- 明确缓存刷新策略与主源故障 fallback 行为，避免旧缓存、旧源数据长期污染前端展示。
- 强化“仅展示可追溯已入库新闻”的规则，确保详情、Ask AI、Interpret 等链路都有稳定数据来源。
- 收口本地转发服务的连接管理、时区序列化、错误恢复与健康检查行为。

影响接口或页面：
- `/api/v1/news*`
- 本地 `pobo-proxy` 转发服务
- Dashboard 新闻流与 7x24 视图

验收方式：
- `pobo-proxy` 开启时，最新新闻可稳定进入前端列表与详情页。
- 关闭主源后，系统回退行为明确且不导致页面崩溃。
- 上海时区下，新闻发布时间与源端业务时间一致。

### 3.3 B3 Dashboard 信息架构收口
目标结果：
- Dashboard 资讯入口收敛为当前实际有效的信息结构，避免原始计划与现状不一致。
- 用户可以从列表、热榜、详情、Ask AI 之间顺畅跳转，不出现重复入口和概念混乱。

主要改动方向：
- 以 `Events + 7x24` 为主链路重新梳理资讯入口层级。
- 热榜、筛选、详情、Ask AI 的入口关系统一，减少重复卡片和歧义按钮。
- 清理文档与实现中仍残留的 `Raw News` 旧口径，避免后续实施偏航。

影响接口或页面：
- Dashboard 资讯主区域
- 热榜入口与详情弹窗
- Ask AI 跳转链路

验收方式：
- 用户能够在 Dashboard 内完成“发现资讯 -> 查看详情 -> Ask AI”的完整路径。
- 页面中不再同时存在冲突或语义重复的资讯入口。

### 3.4 B4 Insight / 简报一致性
目标结果：
- 简报内容与当前新闻源一致，不再混用旧源、旧缓存或旧语言输出。
- 早报/晚报切换、展示时间、内容时间窗与用户设置保持一致。
- 简报与新闻详情、Dashboard 列表之间具备可追溯关系。

主要改动方向：
- 统一简报生成链路中的新闻来源、语言要求和时间窗过滤逻辑。
- 补齐简报展示层的时区转换与会话切分一致性，避免用户设置变更后内容与时间展示脱节。
- 增加简报详情与资讯链路的关联能力，确保用户能追溯简报引用的新闻上下文。

影响接口或页面：
- `/api/v1/insights/*`
- `AIDailyBriefCard`
- Insight 详情弹窗

验收方式：
- 简报内容语言与当前源数据一致。
- 上海时区下，简报时间展示、早晚切换和新闻时间窗行为一致。

### 3.5 B5 Docker 验收与运维闭环
目标结果：
- 第二阶段当前全部能力在 Docker 部署下具备可重复验收路径。
- 运行态问题进入文档化验收，不再依赖临时排障经验。

主要改动方向：
- 补充 Docker 环境下的新闻源连通性检查、缓存刷新步骤、数据库迁移检查点。
- 明确镜像构建、容器启动、主源联通、详情打开、Ask AI、简报查看的手工验收顺序。
- 将当前已暴露的问题纳入正式验收清单，如旧缓存残留、主源时间语义、详情快照回退。

影响接口或页面：
- Docker Compose 启动链路
- 后端迁移与缓存
- Dashboard / Insight / News 主流程

验收方式：
- `docker compose up -d` 后可完成完整主链路验收。
- 不依赖手工改库即可验证新闻列表、详情、Ask AI、简报和事件链路。

## 4. 接口与兼容性基线
### 4.1 保留的第二阶段接口基线
- `GET /api/v1/news/hot-rank`
- `GET /api/v1/news/by-sector/{sector}`
- `GET /api/v1/news/by-topic/{topic}`
- `POST /api/v1/news/{article_id}/ask`
- `/api/v1/insights/*`

### 4.2 不再作为 B 阶段主路径的能力
- 不再规划独立 `/api/v1/briefs/*` 作为第二阶段 B 的正式实现目标。

### 4.3 兼容性要求
- 第一阶段 `events / marker / deeplink` 行为不得回归。
- 第二阶段 A 已完成能力不得回退。
- 当前 `pobo-proxy` 主源链路需保持可用，并与现有 fallback 机制兼容。

## 5. B 阶段验收标准
### 5.1 后端回归
- 新闻详情接口在主源、快照、缓存三类路径下均可返回完整可读数据。
- `pobo-proxy` 时间字段在系统内部统一为可正确换算的 UTC 表达。
- `Ask AI`、热榜、按标签查询在主源切换后不出现 404 或空上下文。

### 5.2 前端回归
- 新闻详情打开后摘要、标题、时间、来源字段完整可读。
- Dashboard 可从资讯列表进入详情，再进入 Ask AI。
- 简报时间与新闻时间展示均遵循上海时区设置。

### 5.3 Docker 手工验收
- `docker compose up -d` 后，新闻列表、详情、Ask AI、简报查看主链路可用。
- `pobo-proxy` 为主源时，前端时间展示与上海时区一致。
- 缓存清理或缓存过期后，旧错误数据不会持续污染前端展示。

### 5.4 单列检查项
- 需要将“缓存导致旧数据残留”的验证点单列纳入验收步骤。
- 需要将“主源关闭后的 fallback 行为”单列纳入验收步骤。

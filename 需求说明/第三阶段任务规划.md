# 第三阶段任务规划（深度分析与平台成熟度）

## 1. 文档说明
- 对比来源：`第二阶段任务规划-B.md` 与 `第二阶段已完成任务-B.md` + 当前代码基线。
- 统计口径：以 git 提交记录与代码实际落地结果为基线，结合第二阶段 B 的剩余收口项。

## 3. 任务分组

### 3.1 C1 第二阶段收口（前置任务）

**目标结果**：
- 完成 B3/B5 剩余项，确保第二阶段全量可验收后再进入第三阶段建设。

**主要改动方向**：
- 清理 Dashboard 中 `hotNewsItems` 旧查询链路，统一走 `hot-rank` 接口，消除信息架构歧义。
- 固化 Docker 发布验收清单：主源开启/关闭 fallback 行为、缓存清理后数据一致性、全链路巡检顺序。
- 补齐热榜→详情→Ask AI 页面级交互一致性回归用例。

**影响范围**：
- Dashboard 数据层（`useDashboardData`）
- Docker Compose 启动链路
- 后端回归测试用例

**验收方式**：
- Docker 启动后，按固化清单执行全链路验收，所有检查项通过。
- Dashboard 不再存在 `hotNewsItems` 与 `hot-rank` 双路径数据不一致问题。

### 3.2 C2 研报生成引擎

**目标结果**：
- 用户可对单只股票/板块/品种发起 AI 研报生成，获得结构化的深度分析报告。
- 研报内容至少覆盖：基本面概览、近期事件与影响、技术面简析、风险提示。
- 研报支持历史回看与导出。

**主要改动方向**：

#### 3.2.1 研报数据模型
- 新增 `research_reports` 表：`id, user_id, symbol, report_type (stock/sector/commodity), title, content (JSONB: sections), status (pending/running/completed/failed), generated_at, error_message`。
- 研报内容结构：`headline → summary → sections[] (title + body + charts[]) → risk_factors → disclaimer`.

#### 3.2.2 研报生成服务
- 新增 `ResearchReportService`，复用 PTC agent 进行深度分析：
  - 拉取标的基本面数据（FMP / YFinance）
  - 拉取近期相关新闻与事件
  - 拉取技术面数据（OHLCV + 技术指标）
  - 由 LLM 综合生成结构化研报
- 支持手动触发 + 定时自动生成（复用 AutomationScheduler）
- 生成超时控制（默认 300s），失败可重试

#### 3.2.3 研报 API
- `POST /api/v1/research/reports` — 手动生成研报（body: `{symbol, report_type}`，返回 `202` + `report_id`）
- `GET /api/v1/research/reports` — 研报列表（按 symbol/type 筛选，分页）
- `GET /api/v1/research/reports/{id}` — 研报详情（含完整 sections）
- `POST /api/v1/research/reports/{id}/retry` — 重试失败研报
- `DELETE /api/v1/research/reports/{id}` — 删除研报

#### 3.2.4 前端研报页面
- Dashboard 新增"研报"入口，展示研报列表（卡片：symbol + 标题 + 时间 + 状态）。
- 研报详情页：结构化渲染（标题 → 摘要 → 分节内容 → 风险提示），支持 Markdown + 图表。
- 生成入口：自选/组合列表中增加"生成研报"按钮，支持选择 stock/sector/commodity 类型。
- 状态流转：`pending` → 骨架屏 → `completed` → 内容渲染 / `failed` → 错误提示 + 重试按钮。

**影响接口或页面**：
- 新增 `/api/v1/research/*` 路由
- 新增 `ResearchReport` 数据库模块与迁移
- Dashboard 新增研报入口与详情页面
- PTC agent 新增研报子代理（`research-report` subagent）

**验收方式**：
- 对任意已覆盖 symbol 发起研报生成，3 分钟内返回结构化报告。
- 报告包含基本面、事件、技术面三个以上有实质内容的章节。
- 生成失败时状态可见、可重试。
- Docker 环境下全链路可用。

### 3.3 C3 事件影响分析

**目标结果**：
- 事件与价格走势建立量化关联，用户可查看事件发生后标的的价格变动。
- 支持"事件→影响"回溯查询，形成事件驱动分析的闭环。

**主要改动方向**：

#### 3.3.1 事件影响量化
- 新增 `event_impact` 计算模块：
  - 事件发生后 N 分钟/小时/日的价格变动（基于 OHLCV 缓存）
  - 关联标的涨跌方向与事件 importance 的对照
  - 同类事件的历史影响统计（按 sector + topic 聚合）
- 影响数据在事件入库时异步计算，结果存入 `market_events.event_impact` JSONB 字段

#### 3.3.2 影响分析 API
- `GET /api/v1/events/{event_id}/impact` — 事件影响详情（价格变动 + 同类对比）
- `GET /api/v1/events/impact-summary?symbol=X&days=N` — 标的历史事件影响汇总

#### 3.3.3 前端展示
- EventDetailModal 增加"影响分析"标签页：价格走势小图 + 影响指标 + 同类对比。
- MarketView 事件 marker 增加影响方向标识（涨/跌/平 + 幅度）。

**影响接口或页面**：
- `market_events` 表新增 `event_impact` JSONB 字段
- EventService 新增影响计算逻辑
- EventDetailModal、MarketView marker 组件

**验收方式**：
- 点击任意已完成影响计算的事件，可查看关联标的价格变动数据。
- 有对标的事件，同类对比数据可展示。
- 事件 marker 正确展示涨跌方向。

### 3.4 C4 投资组合智能

**目标结果**：
- 投资组合与自选列表成为个性化分析的数据源，而非仅静态记录。
- 用户可获得组合层面的风险提示、集中度分析、相关新闻聚合。

**主要改动方向**：

#### 3.4.1 组合风险分析
- 新增 `PortfolioAnalytics` 模块：
  - 持仓集中度（单标的/板块/行业占比）
  - 组合层面相关性矩阵（基于历史价格数据）
  - 简单 VaR 估算
- 计算在组合变更时异步触发，结果缓存至 Redis（TTL 1h）

#### 3.4.2 组合感知信息流
- 新闻/事件列表增加"仅看持仓相关"筛选模式。
- Insight 生成时纳入用户 Top 持仓标的（现有 `DEFAULT_MAX_SYMBOLS` 已预留扩展点）。
- Dashboard 概览卡片增加组合健康度指标（集中度警告、重大事件提醒）。

#### 3.4.3 API 扩展
- `GET /api/v1/users/me/portfolio/analytics` — 组合分析概览
- `GET /api/v1/users/me/portfolio/risks` — 风险明细
- `GET /api/v1/users/me/portfolio/news` — 持仓相关新闻聚合

**影响接口或页面**：
- 新增组合分析 API
- Dashboard 概览卡片、新闻筛选、Insight 个性化
- 前端新增组合分析面板

**验收方式**：
- 持仓用户可在 Dashboard 看到组合层面的集中度与风险指标。
- "仅看持仓"筛选可正确过滤新闻列表。
- 组合分析数据在持仓变更后自动刷新。

### 3.5 C5 自定义看板

**目标结果**：
- 用户可自定义 Dashboard 布局，选择展示哪些卡片及其顺序。
- 不同用户角色（短线交易/中长期投资/行业研究）可获得差异化的默认布局。

**主要改动方向**：

#### 3.5.1 看板配置模型
- `user_preferences` 新增 `dashboard_layout` JSONB 字段：
  - `cards[]`: 卡片类型 + 排序 + 尺寸 (sm/md/lg) + 可见性
  - 预设模板：`trader` / `investor` / `researcher`
- 前端从偏好中读取布局配置，未配置时使用默认布局

#### 3.5.2 卡片组件化
- Dashboard 卡片注册为可插拔组件：
  - 现有卡片：`WatchlistCard`, `PortfolioSummaryCard`, `EventsCard`, `NewsFeedCard`, `HotRankCard`, `InsightCard`, `MarketOverviewCard`
  - 新增卡片：`ResearchReportCard`, `PortfolioRiskCard`, `EconomicCalendarCard`
- 卡片支持拖拽排序（react-dnd 或原生实现）

#### 3.5.3 前端实现
- Dashboard 进入"编辑模式"后可拖拽排序、显示/隐藏卡片、调整尺寸。
- 布局变更自动保存至后端偏好。
- 预设模板一键切换。

**影响接口或页面**：
- `PUT /api/v1/users/me/preferences` 扩展 `dashboard_layout`
- Dashboard 页面架构重构为可插拔卡片系统
- 新增多个卡片组件

**验收方式**：
- 用户可拖拽调整卡片顺序，刷新后保持。
- 切换预设模板后布局即时变化。
- 新增/移除卡片行为正确。

### 3.6 C6 自动化增强

**目标结果**：
- 自动化从"定时+价格触发"扩展到"事件触发+研报定时生成+多条件组合"。
- 自动化执行结果可追溯、可通知。

**主要改动方向**：

#### 3.6.1 新增触发类型
- **事件触发**：指定 symbol 出现高重要性事件时自动推送/生成研报
- **研报定时**：每日/每周定时对自选标的生成研报
- **组合条件**：支持 AND/OR 组合多个条件（如"价格跌破 X AND 出现负面事件"）

#### 3.6.2 通知渠道
- 自动化执行结果支持 Webhook 推送（已有 `webhook_client.py` 基础）。
- 前端站内通知（自动化执行完成/失败的状态提示）。

#### 3.6.3 后端扩展
- `automations` 表扩展 `trigger_config` 支持复合条件。
- `AutomationScheduler` 扩展事件触发检查（订阅 EventService 事件）。
- 自动化执行结果页增强：展示研报/分析内容摘要，而非仅执行状态。

**影响接口或页面**：
- 自动化 CRUD API 扩展触发类型
- AutomationScheduler / PriceMonitorService 扩展
- 前端自动化创建表单与执行历史页

**验收方式**：
- 可创建"事件触发"自动化，在高重要性事件发生时收到通知。
- 可创建"每周研报"自动化，定时对自选标的生成研报。
- 组合条件自动化正确判断 AND/OR 逻辑。

### 3.7 C7 生产化与运维成熟度

**目标结果**：
- 系统具备生产级可观测性、错误恢复与运维文档。
- 第三阶段全部能力在 Docker 部署下可重复验收。

**主要改动方向**：

#### 3.7.1 可观测性
- 结构化日志完善：关键服务（EventService, InsightService, ResearchReportService）输出统一格式的运行时指标。
- 健康检查增强：`/health` 增加新闻源连通性、Redis 状态、数据库连接池状态检查项。
- 慢查询与异常率监控（日志级别控制，不引入额外基础设施）。

#### 3.7.2 错误恢复
- 新闻源 fallback 链路统一：pobo-proxy → ginlix-data → FMP → yfinance，每层切换记录日志。
- 研报生成失败自动重试策略（最多 2 次，间隔 30s）。
- 事件聚合服务异常自恢复（重启后从上次成功游标继续）。

#### 3.7.3 Docker 验收清单固化
- 在 B5 基础上补充第三阶段验收项：
  - 研报生成链路（触发 → 生成 → 查看）
  - 事件影响计算链路
  - 组合分析链路
  - 自定义看板布局持久化
  - 自动化增强链路
- 形成 `验收清单-第三阶段.md`，作为发布准入标准。

**影响接口或页面**：
- `/health` 端点
- 各服务日志输出
- Docker Compose 配置

**验收方式**：
- `/health` 返回所有关键依赖状态。
- 任一新闻源故障时，系统自动降级且日志可追踪。
- 按清单执行全链路验收，所有检查项通过。

## 6. 实施优先级与建议顺序

```
C1（第二阶段收口） → C2（研报引擎） → C3（事件影响） → C4（组合智能） → C5（自定义看板） → C6（自动化增强） → C7（生产化收口）
```

**排序理由**：
- **C1 优先**：确保第二阶段全量可验收后再进入新建设，避免欠债累积。
- **C2 其次**：研报是第三阶段核心价值交付，也是最重的模块，尽早启动以留出打磨时间。
- **C3 紧随**：事件影响分析依赖 C2 的 PTC agent 经验，且与 C2 共享事件/新闻数据源。
- **C4 后置**：组合智能依赖 C2/C3 的分析能力，需要前两者稳定后再做集成。
- **C5/C6 并行**：自定义看板与自动化增强相对独立，可在 C4 完成后并行推进。
- **C7 持续**：生产化改进贯穿整个阶段，最终在 C6 完成后统一收口。

## 7. 明确不进入第三阶段的内容

- TTS（文本转语音播报）
- 地图能力（产业链地图/地域风险热力图）

## 8. 关键风险与假设

### 8.1 假设
- pobo-proxy 国内新闻源在第三阶段持续可用，新闻覆盖率不低于当前水平。
- Qwen 模型调用配额与延迟满足研报生成需求（单次研报预计 5-8 次 LLM 调用）。
- 当前 Daytona 沙箱 + PTC agent 架构可支撑研报生成的计算需求。

### 8.2 风险
- **LLM 成本**：研报生成单次调用量大，需关注 token 消耗与成本控制，建议增加用户级配额。
- **生成质量**：LLM 生成的研报质量依赖输入数据完整性，需重点关注数据缺失场景的兜底策略。
- **性能**：事件影响计算如果全量回填历史数据，首次运行可能有较大计算量，需分批处理。
